When I listed the tools integrated into InboxGuard, a few people asked if I was joking.

21 tools. For an app that deletes emails.

The answer isn't "this project needs them." The answer is: **I needed to learn them, and this was the safest place to do it.**

Security tooling has a steep learning curve. Misconfigured SARIF uploads, false-positive suppression, custom Semgrep rules, OWASP ZAP authentication — these things take time to figure out. The cost of figuring them out on a production system is downtime and incidents. On a Gmail cleanup tool, the cost is nothing. The perfect environment to build the muscle.

## How they connect

The layering is the point. Each layer catches what the previous one misses.

### Layer 1 — at the keyboard

**ESLint** and **Prettier** run in the editor and on save. They catch the most obvious issues before you even think about committing — unused imports, shadowed variables, inconsistent formatting. This is the cheapest possible feedback loop.

### Layer 2 — at the commit boundary

**Husky** manages the Git hooks. Two hooks run on every `git commit`:

- `pre-commit`: runs **ESLint** (with `--max-warnings 0`), **Prettier** (check mode), and **Gitleaks** (secret scanning)
- `commit-msg`: enforces **Conventional Commits** format — `feat:`, `fix:`, `chore:`, etc.

This is bypassable with `--no-verify`, but that's intentional. You shouldn't be blocked from committing when you need to checkpoint work in progress. The next layers still catch anything that slips through.

### Layer 3 — in CI on every push

Four parallel jobs:

| Job | Tools | What it catches |
|---|---|---|
| `backend` | Maven, JUnit | Compilation errors, unit test failures, ArchUnit violations |
| `frontend` | Node, ESLint, TypeScript | Type errors, lint violations, build failures |
| `security` | **Trivy** + **Semgrep** + **Reviewdog** | CVEs in deps, custom rule violations, inline PR comments |
| `analyze` | **CodeQL** | Semantic data-flow vulnerabilities across Java + TypeScript |

**Trivy** scans three surfaces: `pom.xml`/`package.json` for CVEs, the filesystem for hardcoded secrets, and the CI config for IaC misconfigurations. Output is uploaded as SARIF to GitHub's Security tab.

**Semgrep** runs six custom project rules — the ones that enforce InboxGuard's specific invariants, like "Gmail SDK types must not appear outside `GmailAdapter`" and "no hardcoded AES keys."

**Reviewdog** picks up both ESLint and Semgrep output and posts them as inline comments on the PR diff rather than just failing the build. Much easier to act on.

**CodeQL** builds a graph of the entire codebase and runs dataflow queries. It can trace a tainted input from an HTTP controller parameter through a service layer and into a database call — across method boundaries and class hierarchies. Semgrep can't do this.

### Layer 4 — the live application

**OWASP ZAP** runs in its own workflow. It:

1. Starts a PostgreSQL container
2. Builds and runs Spring Boot with the `test` profile
3. Waits for `/actuator/health` to return 200
4. Runs a full authenticated scan — checking for missing security headers, CSRF gaps, injection points, and authentication bypass

This catches things that static analysis physically cannot. A header being set to the wrong value. A CSRF token that validates correctly in code but gets stripped by a middleware. Auth that works in unit tests but breaks at the HTTP layer.

### Layer 5 — pull request policy

**Danger JS** enforces policies that can't be expressed as test failures:

- PRs over 800 lines get a warning ("consider splitting")
- New service files require matching test files
- Changes to `SecurityConfig.java` get an explicit review annotation
- Files named `credentials.json` or `.env` fail the build outright

**Reviewdog** (which also runs Semgrep inline) posts findings directly on the diff line — not just in the PR comment thread, but on the exact line that triggered the rule.

### Layer 6 — the production server

**Helmet.js** runs in `frontend/server.cjs`, the Express server that serves the built SPA in production. One line sets `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy` on every response.

Spring Security does the same for API responses — matching the secure header baseline across both servers.

## Architecture documentation as a layer

The documentation tools aren't just nice-to-haves — they're part of the enforcement chain.

**Danger JS** fails PRs that add new services or endpoints without updating `docs/architecture/system.md`. **adr-tools** gives every significant decision a numbered record in `docs/adr/`. **Mermaid.js** diagrams render directly in GitHub without any external tooling. **Structurizr Lite** gives the C4-model view for formal architecture review. **D2** handles the complex multi-service diagrams that Mermaid can't layout cleanly.

**GitDiagram** and **GitNexus** are the AI-assistance tools: GitDiagram generates an instant visual overview of the repo structure, GitNexus builds a queryable graph of cross-file dependencies for semantic code navigation.

## What I actually learned

The most useful insight: **security tools generate a lot of noise by default**. The work isn't setting them up — it's tuning them.

Writing the `.gitleaks.toml` exclusions so you don't false-positive on test fixtures. Writing Semgrep rules tight enough to not fire on comments. Getting ZAP to authenticate before scanning so it can actually test protected endpoints.

That tuning is where the real understanding lives. You can only do it when the tool is actually running on something.

The second insight: **layering is multiplicative, not additive**. Each layer isn't just "one more check" — it catches a category of bug that the previous layer is structurally incapable of finding. Gitleaks catches what Semgrep misses. ZAP catches what CodeQL can't reach. Danger JS catches what automated scanners don't understand. Together they cover most of the space.

21 tools for a Gmail cleaner. Every one of them earned its place.
