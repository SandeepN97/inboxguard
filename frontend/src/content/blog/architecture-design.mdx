Good architecture docs answer four questions at increasing zoom levels:

1. **Who uses the system and what external things does it touch?** (System Context)
2. **What deployable pieces make it up?** (Container)
3. **How are the backend layers organized inside one container?** (Component / Hexagonal)
4. **What happens at runtime for the main use case?** (Sequence)

These are Mermaid diagrams — they live next to the code in `docs/architecture/system.md` and render natively on GitHub. The same source is embedded here.

Key decisions visible at this level:
- **Single user** — no multi-tenancy, no auth hierarchy, no team model
- **Gmail is external** — the system never stores email content, only message IDs and metadata
- **PostgreSQL is external** — rules and run history survive restarts

Security decisions visible here:
- **XSRF-TOKEN cookie** — Spring Security's `CookieCsrfTokenRepository`; Axios echoes it as `X-XSRF-TOKEN` header on every mutating request
- **No email content stored** — `oauth_tokens` stores only the AES-256-GCM encrypted refresh token
- **Flyway** — schema migrations are versioned and auditable, never auto-DDL in prod

/>

The key invariants ArchUnit enforces:
- `..domain..` → imports nothing from `..infrastructure..`, `..web..`, Spring, JPA, or Google
- `..application..` → imports nothing from `..infrastructure..` or `..web..`
- `com.google.api.services.gmail.*` → only appears in `..infrastructure.gmail..`
- `jakarta.persistence.*` → never appears in `..domain..`

These run as tests in CI. A violation breaks the build.

## 4 — Cleanup Run Sequence

The main use case at runtime. `POST /api/runs` returns immediately (202); the Gmail I/O runs on a virtual thread.

<Mermaid
  caption="Sequence — async cleanup run using Project Loom virtual threads"
  chart={`
sequenceDiagram
  actor U as Admin
  participant SPA as React SPA
  participant API as Spring Boot
  participant VT as Virtual Thread
  participant DB as PostgreSQL
  participant G as Gmail API

  U->>SPA: Click "Run Cleanup" (LIVE)
  SPA->>API: POST /api/runs {mode: LIVE}
  API->>DB: INSERT cleanup_runs status=RUNNING
  API->>VT: Thread.ofVirtual().start()
  API-->>SPA: 202 Accepted {id, status: RUNNING}

  loop poll every 3 s while RUNNING
    SPA->>API: GET /api/runs/{id}
    API-->>SPA: {status: RUNNING, emailsProcessed: 0}
  end

  loop for each active SenderRule
    VT->>G: messages.list q=from:sender
    G-->>VT: message IDs
    VT->>G: messages.batchModify remove INBOX label
    G-->>VT: 200 OK
    VT->>DB: INSERT cleanup_run_results
  end

  VT->>DB: UPDATE cleanup_runs status=COMPLETED
  SPA->>API: GET /api/runs/{id}
  API-->>SPA: {status: COMPLETED, emailsProcessed: 142, emailsDeleted: 142}
  SPA-->>U: Show results
`}
/>

Why virtual threads here?

The Gmail API calls are blocking I/O — each `messages.list` + `messages.batchModify` pair waits for an HTTP round trip. With Project Loom, Spring Boot parks the virtual thread during the wait and reuses the carrier thread for other work. No reactive programming, no callback hell, no `CompletableFuture` chains — just blocking code that scales.

## 5 — OAuth2 + Token Storage Flow

How the Gmail refresh token gets stored securely on first login.

<Mermaid
  caption="OAuth2 flow — refresh token encrypted with AES-256-GCM before DB storage"
  chart={`
sequenceDiagram
  actor U as Admin
  participant SPA as React SPA
  participant API as Spring Boot
  participant G as Google OAuth2
  participant ENC as TokenEncryptionService
  participant DB as PostgreSQL

  U->>SPA: Click "Continue with Google"
  SPA->>API: GET /api/auth/google
  API-->>SPA: 302 → /oauth2/authorization/google
  SPA->>G: Authorization request (scope: gmail.modify)
  G-->>U: Consent screen
  U->>G: Approve
  G-->>API: Authorization code + refresh token
  API->>ENC: encrypt(refreshToken, AES-256-GCM key)
  ENC-->>API: base64(nonce ‖ ciphertext ‖ tag)
  API->>DB: INSERT oauth_tokens (encrypted_token)
  API-->>SPA: Session cookie set, redirect to /?signing_in=1
  SPA->>API: GET /api/auth/me
  API-->>SPA: {email, name, picture}
`}
/>

The AES-256-GCM key (`GMAIL_TOKEN_ENCRYPTION_KEY`) is a 32-byte secret injected at runtime via environment variable. It is never stored in source, never in the database, never logged. If it's absent, the application refuses to start.
